# Harn core compatibility spike — 2026-07-17

## Result

Accept `harn_agent.Agent` as the initial AIBB tool-loop engine, pinned through `uv.lock`. Do not use the Harn coding-agent CLI or high-level session lifecycle. Pi remains a contingency, not a second implementation target.

## Proven

- The provider receives the exact AIBB system prompt and only explicitly constructed tool schemas.
- Global `~/.harn` material and project `AGENTS.md` files do not enter the low-level agent context.
- A Harn tool wrapper can call a separately launched, official-protocol MCP stdio server and return its result to the next model turn.
- Tool execution is sequential.
- Harn tool/message lifecycle events are observable for AIBB recording.
- A curator message queued while a tool is running is delivered once at the next model boundary.
- Model-visible messages, response IDs, context generation, and an AIBB-owned opaque provider-state envelope serialize and reconstruct exactly at a safe checkpoint.
- The private store appends hash-chained JSONL events and atomically replaces a checkpoint bound to the current event hash.
- The low-level agent makes no automatic follow-up or compaction call.

## Not yet proven

- Fidelity of any real provider adapter's raw responses, encrypted reasoning items, native continuation identifiers, retries, or token accounting.
- Recovery from process death during an in-flight provider stream or MCP mutation; the spike covers reconstruction from the last durable safe checkpoint.
- The production TUI and its crash/reconnect behavior.
- AIBB's explicit compaction strategies. Only absence of Harn automatic compaction is proven.
